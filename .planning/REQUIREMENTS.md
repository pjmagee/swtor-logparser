# Requirements: SWTOR Log Parser — v1.1 WinUI 3 Overlay & Dev Tooling

**Defined:** 2026-06-12
**Core Value:** The live DPS/HPS stats pipeline stays correct and reliable while the codebase becomes safe to maintain and extend — no regressions to parsing or the reactive stream.

> v1.1 is a **host re-implementation + tooling refresh**, not new product capability. The core library `SwtorLogParser` and the live `CombatLogsMonitor.Instance.DpsHps` stream are **frozen**. Every requirement below changes a host, the test project, interop, or tooling — never the parser or the stream.

## v1 Requirements

Requirements for this milestone. Each maps to a roadmap phase.

### Overlay (OVL) — WinUI 3 host replacing WinForms

- [x] **OVL-01**: A WinUI 3 (Windows App SDK) overlay project exists as an **unpackaged, self-contained** app that launches a window from a published `.exe` — and adding it keeps the `.slnx` build and the Native AOT CLI publish green (AOT/CI guardrails intact)
- [x] **OVL-02**: The overlay subscribes to `CombatLogsMonitor.Instance.DpsHps` and renders live per-player rows (Player / DPS / Crit% / HPS / Crit%) updated on the UI dispatcher with no cross-thread crash, reusing the core `View` sliding-expiry types unchanged
- [ ] **OVL-03**: The WinUI 3 transparency + click-through approach is validated by a throwaway **spike** on a real window before the production grid port (de-risks the highest-uncertainty work)
- [ ] **OVL-04**: The overlay is transparent and borderless, at visual parity with the WinForms overlay
- [ ] **OVL-05**: The user can drag the overlay to reposition it
- [ ] **OVL-06**: The user can toggle click-through mode (mouse passes through to the game); default is **off** (overlay movable/interactive)
- [~] **OVL-07**: The overlay persists window position, size, and opacity across runs (local settings file; no core dependency) — **position + size DONE in Phase 9** (`%LocalAppData%\SwtorLogParser\settings.json`, source-gen JSON, corruption-safe); **opacity persistence DEFERRED to Phase 10 per D-04** (reserved `OverlaySettings.Opacity` field exists, not yet applied)
- [x] **OVL-08**: The user can increase/decrease the overlay font size, at parity with the WinForms controls
- [ ] **OVL-09**: After WinUI 3 parity is validated (transparent, draggable, live render, topmost over Fullscreen-Windowed), the WinForms overlay is removed and WinUI 3 is the **sole** overlay host

### Interop (INT) — CsWin32 + topmost

- [ ] **INT-01**: All overlay Win32 interop is generated by the **Microsoft.Windows.CsWin32** source generator (driven by `NativeMethods.txt`), replacing the hand-written `NativeMethods.cs` P/Invoke — closes #3
- [ ] **INT-02**: The overlay stays on top of SWTOR running **Fullscreen (Windowed)/Borderless** by re-asserting `HWND_TOPMOST` on foreground changes (`SetWinEventHook` + `SetWindowPos`) — closes BL-01
- [ ] **INT-03**: The overlay does not steal focus from the game and stays out of Alt-Tab (`WS_EX_NOACTIVATE` / tool-window styles)

### Testing (TEST) — xUnit → MSTest SDK

- [ ] **TEST-01**: The test project is migrated from xUnit to the **MSTest .NET SDK** (`MSTest.Sdk`); all 106 tests pass with `InternalsVisibleTo` internals access and source-swap serialization intact — closes #2
- [ ] **TEST-02**: CI stays green and **code coverage is preserved** after the migration (e.g. `UseVSTest=true`), with the workflow updated in the same change

### Dev Tooling (DX) — VSCode

- [ ] **DX-01**: VSCode `launch.json` provides an F5 debug configuration for every runnable host (managed CLI, Native CLI, WinUI 3 overlay)
- [ ] **DX-02**: VSCode `tasks.json` provides build, test, and Native-AOT-publish tasks

### Docs (DOCS)

- [ ] **DOCS-01**: The README/docs are refreshed to describe the WinUI 3 overlay, .NET 10, the three hosts, the run/debug story, and the **windowed-borderless vs exclusive-fullscreen** limitation

## Out of Scope

Explicitly excluded. Documented to prevent scope creep.

| Feature | Reason |
|---------|--------|
| Overlay over **exclusive fullscreen** | Architecturally impossible for a normal window without DirectX injection; document the windowed-borderless requirement instead |
| Changes to the parser, domain model, or live DPS/HPS stream | Core is validated and frozen; v1.1 swaps the overlay host + tooling only |
| New stats / metrics / new hosts | v1.1 is re-implementation + tooling, not new product capability |
| WinUI 3 overlay on Native AOT | AOT applies only to `Native.Cli`; the overlay is a normal managed JIT host |
| Cross-platform / Linux support | App is intentionally Windows-only; WinUI 3 + Win32 interop reinforce this |
| MAUI as the UI framework | User chose WinUI 3 (issue #4); MAUI's better AOT story is moot since the overlay isn't AOT-constrained |

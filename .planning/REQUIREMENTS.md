# Requirements: SWTOR Log Parser — v1.1 WinUI 3 Overlay & Dev Tooling

**Defined:** 2026-06-12
**Core Value:** The live DPS/HPS stats pipeline stays correct and reliable while the codebase becomes safe to maintain and extend — no regressions to parsing or the reactive stream.

> v1.1 is a **host re-implementation + tooling refresh**, not new product capability. The core library `SwtorLogParser` and the live `CombatLogsMonitor.Instance.DpsHps` stream are **frozen**. Every requirement below changes a host, the test project, interop, or tooling — never the parser or the stream.

## v1 Requirements

Requirements for this milestone. Each maps to a roadmap phase.

### Overlay (OVL) — modern transparent overlay replacing WinForms

> **PIVOT (2026-06-12): WinUI 3 → Dear ImGui.** Phase 8/9 built a WinUI 3 overlay, but WinUI 3 cannot do **clear per-pixel transparency** (`WS_EX_LAYERED` blanks its DirectComposition swapchain; only frosted acrylic/mica are see-through). The overlay was pivoted to an immediate-mode **Silk.NET/OpenGL + Dear ImGui** host (`SwtorLogParser.Overlay.ImGui`) with a GLFW `TransparentFramebuffer` — genuine clear transparency, the correct tool for a game overlay. The WinUI 3 overlay was deleted alongside WinForms. The OVL requirements below are satisfied by the ImGui overlay.

- [x] **OVL-01**: A self-contained overlay project exists and launches from a published `.exe` (WinUI 3 scaffold shipped Phase 8, then superseded by the ImGui overlay — self-contained win-x64 publish)
- [x] **OVL-02**: The overlay subscribes to `CombatLogsMonitor.Instance.DpsHps` and renders live per-player rows (Player / DPS / Crit% / HPS / Crit%), DPS-descending, reusing the core `View` sliding-expiry types unchanged — immediate-mode each frame, no cross-thread crash
- [x] **OVL-03**: The transparency approach was de-risked (the ImGui pivot itself proved clear transparency via GLFW `TransparentFramebuffer` after the WinUI layered-window approach failed)
- [x] **OVL-04**: The overlay is transparent and borderless — **clear** see-through (better than the WinForms color-key), with an opacity slider
- [x] **OVL-05**: The user can drag the overlay to reposition it (☰ grip, manual cursor-delta drag)
- [ ] **OVL-06**: ~~Click-through toggle~~ — **DEFERRED** (`WS_EX_TRANSPARENT` alone doesn't pass clicks through the GLFW window; removed per user — small movable overlay doesn't need it; revisit with a layered/mouse-passthrough approach)
- [x] **OVL-07**: The overlay persists **position + opacity + font** across runs (`%LocalAppData%\SwtorLogParser\imgui-overlay.json`, source-gen JSON, corruption-safe). Window size is content/auto, not persisted.
- [x] **OVL-08**: The user can increase/decrease the overlay font size (+/- buttons, persisted)
- [x] **OVL-09**: **BOTH** predecessor overlays removed (WinForms `SwtorLogParser.Overlay` **and** the WinUI 3 detour); the ImGui overlay is the **sole** overlay host — `.slnx` builds Release green, 116/116 tests pass

**Bonus (new feature, beyond original scope):** the overlay has a **mini combat log** — a `Log` toggle that expands a rolling, human-readable ability/damage feed (`time  player  ability -> target  amount (crit)`), GUIDs/syntax stripped, fed from the core `CombatLogChanged` event (no core changes). Useful for streamers sharing their rotation.

## Out of Scope

Explicitly excluded. Documented to prevent scope creep.

| Feature | Reason |
|---------|--------|
| Overlay over **exclusive fullscreen** | Architecturally impossible for a normal window without DirectX injection; document the windowed-borderless requirement instead |
| Changes to the parser, domain model, or live DPS/HPS stream | Core is validated and frozen; v1.1 swaps the overlay host + tooling only |
| New stats / metrics / new hosts | v1.1 is re-implementation + tooling, not new product capability |
| WinUI 3 overlay on Native AOT | AOT applies only to `Native.Cli`; the overlay is a normal managed JIT host |
| Cross-platform / Linux support | App is intentionally Windows-only; WinUI 3 + Win32 interop reinforce this |
| MAUI as the UI framework | User chose WinUI 3 (issue #4); MAUI's better AOT story is moot since the overlay isn't AOT-constrained |

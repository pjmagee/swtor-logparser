<objective>
Implement RFCT-03: re-key the static parse caches from `ReadOnlyMemory<char>.GetHashCode()` (reference+index+length, NOT content) to string content keys (`rom.ToString()`); bound the caches with a size cap + simple insertion-order eviction (a small AOT-safe thread-safe cache implemented in-repo, NO new package); and FIX the latent `Ability`/`GameObject` shared-cache type bug by giving each concrete type its OWN cache.

Purpose: The current key is not content-based, so identical-content roms with different backing memory are not deduped (latent ME-02). Switching to content keys is correct but makes cross-type key collisions MORE likely — which would SURFACE the existing `(Ability?)value` cast collision (a `GameObject` cached under a content key would throw/mis-return when read as `Ability`). Separate per-type caches eliminate the collision and must land TOGETHER with the content-key change to avoid introducing a regression mid-refactor. Bounding caps unbounded memory growth from high-volume log tailing (the only DoS surface).
Output: A `BoundedCache<TValue>` type, three separate content-keyed bounded caches, updated `Parse` methods, and cache tests.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| disk combat log → CombatLogsMonitor → DpsHps | Existing, frozen seam. No NEW untrusted input is introduced by this plan (the overlay only consumes the already-parsed stream). |
| Rx background reader thread → WinUI UI thread | Thread boundary. XAML objects are UI-thread-affine; crossing it incorrectly is a crash, not a security issue, but is the dominant correctness risk. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-01 | Denial of Service | DispatcherQueue flooded by per-OnNext marshaling under dense AoE combat | mitigate | D-01: aggregate off-thread in the locked core list; mirror to Rows only on a 1s DispatcherQueueTimer tick — never marshal per OnNext (caps UI churn regardless of log volume). |
| T-09-02 | Tampering | Cross-thread XAML mutation → COMException 0x8001010E crash | mitigate | D-02 / Pitfall 5: capture the UI DispatcherQueue once on the UI thread; OnNext touches only the internally-locked core list; all Rows mutation happens on the UI-thread timer tick. |
| T-09-03 | Information Disclosure | Player names rendered in the overlay | accept | Player names already appear in the existing WinForms overlay and CLIs; no new disclosure, local display only, no persistence in this plan. |
| T-09-SC | Tampering | npm/pip/cargo installs | accept | No package-manager installs in this plan — the only dependency change is a ProjectReference to the in-repo core library. No external packages added. |
</threat_model>
